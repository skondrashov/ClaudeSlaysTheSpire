## The F11 misroute — verified, and classified

**The player's claim checks out, with one correction.** The wrong `choose 0` was at the **F10→F11 map choice** (idx 184), not on floor 11. Verified from the log:

- The immediately preceding output (idx 183) contains the full map. Floor 11 row: `x1 M, x2 R, x3 E, x5 $, x6 E`. The player's node (F10 x1 M) has `children [1, 2]` → the two legal choices were **x1 Monster (choose 0)** and **x2 Rest (choose 1)** — in plain sight.
- The decision's own reasoning says **"Rest at F11"**; the command was `choose 0`; the translation field reads "→ Monster".
- Cascade confirmed: F11 x1 M has `children [1]` → F12 x1 `?` only. The F12 shop (x2 $) was reachable ONLY via the F11 Rest (x2 R, `children [2]`). One index lost both the rest and the shop. Death with 271 gold, no potions bought, no removal — all downstream of this keystroke.

**Classification: EXECUTION ERROR, not interface gap.** The node list and the children mapping were visible in the prior output; the player had used exactly this index→child mapping correctly four times earlier in the run (including F6→F7, where `choose 0` correctly selected a Rest). The failure was submitting a habitual `choose 0` without re-deriving the index from the children list — a verification skip, the map-navigation twin of chaining `end` on a kill turn. Two mitigations worth flagging anyway (not exculpatory): (a) the harness could echo labeled map options at choice time; (b) a one-line self-check rule ("before any map `choose N`, name the node type you expect and confirm it against the children list") belongs in the same checklist family as the kill-turn rule.

**Magnitude (~10-12 win pts, the run's steepest regret).** Intended line: Rest F11 (27→~49), Shop F12 with 284g (potions — including a possible Weak source — removal, defense), F14 monster entered ~45+, Rest F15 → boss entry ~60-65 with purchased potions. Actual: boss entry 38 with no purchases. The death margin was 1-2 HP on the final turn; the misroute cost ~20-25 HP of entry margin plus the entire shop.

## Disposition findings

**Arithmetic under batch planning — now a cross-run, disposition-class finding.** Five instances in two consecutive runs, all the same shape: a multi-card plan composed as one chained command whose summed energy or summed damage was never re-added before submission.

- Run 227: two 4E-on-3E over-plans (Bash's 2E counted as 1E in both), harness-caught, non-fatal; player self-noted "sum costs explicitly."
- Run 228: two 4E-on-3E over-plans plus one Rampage counter mis-recall (18 vs 13), all at Sentries, **executed and paid for** (~20-25 HP — at A9 that was most of the boss-entry margin).

The per-instance fixes already exist in scattered form (227 self-note; strategy.md kill-turn rule). What's missing is the *kind*: **before submitting any chained multi-card command, re-sum energy and re-derive stateful card values (Rampage counter, upgraded numbers) from the current state, not from memory.** Stateful-value recall and energy summing are exactly the two things batch composition gets wrong. Routed to Curate as one checklist rule, not five anecdotes. (The F11 misroute is arguably the same disposition wearing a map skin: habitual composition outrunning verification.)

**Potion economy: clean this run.** All four acquired potions were spent at correct triggers (two converted a Sentries near-collapse, two were load-bearing in the boss fight). The Run 224/227-era hoarding pattern did not recur. No flag.
